# CloakPipe HTTP API Reference

This document describes the HTTP endpoints that the current server actually registers in source. The route list comes from `crates/cloakpipe-proxy/src/server.rs`, and the behavior details come from `crates/cloakpipe-proxy/src/handlers.rs` and `crates/cloakpipe-proxy/src/tree_handlers.rs`.

## Install CloakPipe

Use the main installer (release binary first, then `cargo install` fallback):

```bash
curl -fsSL https://raw.githubusercontent.com/borgius/cloakpipe/refs/heads/main/install.sh | sh
```

Two quick notes before you copy old examples from elsewhere:

- The current tree routes live under `/tree/...`, not `/v1/tree/...`.
- The current CLI command to start the server is `cloakpipe start`, not `cloakpipe serve`.

## Before you call the API

### Base URL

The server listens on whatever `proxy.listen` is set to in `cloakpipe.toml`.

If you let `cloakpipe start` create a fresh config for you, the generated default is:

- `127.0.0.1:8900`

So the examples in this document use:

- `http://127.0.0.1:8900`

### Required server-side configuration

The proxy authenticates to the upstream LLM provider with the environment variable named by `proxy.api_key_env`.

Examples:

- `OPENAI_API_KEY`
- `OLLAMA_API_KEY`
- `AZURE_OPENAI_API_KEY`

The server can still start if this variable is missing. In that mode, the direct privacy routes remain available:

- `/v1/pseudonymize`
- `/v1/rehydrate`
- `/v1/detect`
- `/v1/vault_stats`
- `/v1/configure`
- `/v1/session_context`

LLM-backed routes such as `/v1/chat/completions`, `/v1/embeddings`, and tree routes that call the upstream model return `503 Service Unavailable` until the variable is set.

The client does **not** need to supply the real upstream provider key to CloakPipe. The current implementation ignores the inbound `Authorization` header and always sends the server-side key configured in the process environment.

That means:

- `curl` calls can omit `Authorization` entirely.
- OpenAI-style SDKs that insist on an API key can use any non-empty placeholder when talking to CloakPipe, unless you put your own auth gateway in front of it.

### Persistence and vault behavior

- Entity-to-token mappings live in the configured vault.
- If `CLOAKPIPE_VAULT_KEY` is missing, CloakPipe generates an ephemeral key at startup.
- With an ephemeral key, mappings do not survive a restart.

### Content types and headers

Most POST endpoints expect `Content-Type: application/json`.

Common request headers:

| Header | Used by | Meaning |
| --- | --- | --- |
| `Content-Type: application/json` | Most POST routes | Standard JSON request body |
| `x-session-id` | `POST /v1/chat/completions` when sessions are enabled | Session key for cross-message context; this is the default header name |
| `OpenAI-Organization` | Proxied chat and embeddings | Passed through to the upstream provider if present |

Common response headers:

| Header | Returned by | Meaning |
| --- | --- | --- |
| `X-CloakPipe-Request-Id` | Chat and embeddings proxy responses | Per-request UUID generated by CloakPipe |
| `X-CloakPipe-Leaked-Entities` | Non-streaming `POST /v1/chat/completions` only | Count of response entities redacted as unexpected leaked PII |

### Error format

Error bodies are not fully uniform.

- Proxied upstream failures are usually passed back with the upstream status code and body.
- Local handler failures often return a plain text string body.
- Successful admin, tree, and session endpoints return JSON.

If you are building a client, treat error bodies as opaque text unless you know you are dealing with a proxied upstream error.

## LLM proxy endpoints

### `GET /health`

Use this to check whether the server is alive.

Health check example:

```bash
curl http://127.0.0.1:8900/health
```

Health check response:

```json
{
  "status": "ok",
  "service": "cloakpipe"
}
```

### `POST /v1/chat/completions`

This is the main endpoint. It accepts an OpenAI-style chat request, masks sensitive text in the prompt, forwards the masked payload to the upstream provider, and then restores tokens in the response.

#### Processing flow

1. Generates a new request ID.
2. Detects whether the request is streaming from `body.stream`.
3. Optionally attaches the request to a session.
4. Pseudonymizes message content before sending it upstream.
5. Calls `{proxy.upstream}/v1/chat/completions` with the server-side API key.
6. Rehydrates the model response before returning it.

For non-streaming responses, it also scans the assistant output for new PII that was **not** already in the vault. If it finds any, it replaces those spans with `[REDACTED]` and sets `X-CloakPipe-Leaked-Entities`.

Chat request body:

The handler accepts arbitrary JSON, but it only pseudonymizes `messages[*].content` when the content is a plain string.

That means the safest request shape is the standard OpenAI chat payload:

```json
{
  "model": "gpt-4o",
  "messages": [
    {
      "role": "user",
      "content": "Email Priya Sharma at priya@example.com and ask about invoice #A-2041."
    }
  ],
  "stream": false
}
```

#### Important limitations

- Only string `messages[].content` values are masked.
- Multimodal message arrays are forwarded untouched.
- Non-streaming rehydration rewrites `choices[].message.content` only.
- Streaming rehydration rewrites `choices[0].delta.content` only.
- The extra leaked-PII scan runs only for non-streaming responses.

#### Session behavior

If `[session].enabled = true`, the handler tries to extract a session ID using `session.id_from`.

Supported modes in the current code:

- `header:<header-name>`
- `connection`

The default config is:

- `header:x-session-id`

With sessions enabled, CloakPipe can:

- keep entity-token mappings consistent across messages,
- resolve simple pronouns and abbreviations,
- track sensitivity escalation for decision-like conversations.

Chat request example:

```bash
curl http://127.0.0.1:8900/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "x-session-id: demo-session-1" \
  -d '{
    "model": "gpt-4o",
    "messages": [
      {
        "role": "user",
        "content": "Summarize the contract terms for Rahul Sharma at rahul@example.com."
      }
    ]
  }'
```

#### Response behavior

##### Non-streaming

- Returns JSON.
- Adds `X-CloakPipe-Request-Id`.
- Adds `X-CloakPipe-Leaked-Entities`.

Example shape:

```json
{
  "id": "chatcmpl-...",
  "object": "chat.completion",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "Rahul Sharma's contract includes ..."
      }
    }
  ]
}
```

##### Streaming

- Returns `Content-Type: text/event-stream`.
- Adds `Cache-Control: no-cache`.
- Adds `X-CloakPipe-Request-Id`.
- Rehydrates text chunk by chunk.

If you already use the OpenAI streaming format, your client can keep consuming SSE the usual way.

#### Upstream error handling

If the upstream provider returns a non-2xx response, CloakPipe forwards that status code and body back to the caller. This is useful because it preserves provider-specific error details.

If CloakPipe itself fails before the upstream call, it returns a local error such as:

- `500 Pseudonymization failed: ...`
- `502 Upstream request failed: ...`

### `POST /v1/embeddings`

This endpoint pseudonymizes embedding input and forwards the request upstream.

#### What it masks

The handler only rewrites the `input` field when it is either:

- a single string, or
- an array of strings.

Other input shapes are forwarded unchanged.

#### What it returns

- The upstream response body is passed through as JSON.
- `X-CloakPipe-Request-Id` is added.

Unlike chat completions, there is no rehydration step on the way back, and the current HTTP handler does not apply ADCPE vector encryption.

Embeddings request example:

```bash
curl http://127.0.0.1:8900/v1/embeddings \
  -H "Content-Type: application/json" \
  -d '{
    "model": "text-embedding-3-small",
    "input": [
      "Priya Sharma lives in Pune.",
      "Contact her at priya@example.com."
    ]
  }'
```

Embeddings response example:

```json
{
  "object": "list",
  "data": [
    {
      "object": "embedding",
      "index": 0,
      "embedding": [0.0123, -0.0456]
    }
  ],
  "model": "text-embedding-3-small"
}
```

## Direct privacy tool endpoints

These routes mirror the six MCP tools over HTTP.

Two practical notes:

- The canonical API paths are under `/v1/...`, and each one also has a root-level alias such as `/pseudonymize`.
- Success responses are normal JSON objects, not JSON serialized into text like the stdio MCP transport.

### `POST /v1/pseudonymize`

Also available at `POST /pseudonymize`.

Request body:

```json
{
  "text": "Send $500 to alice@example.com before June 1, 2026."
}
```

Success response:

```json
{
  "text": "Send AMOUNT_1 to EMAIL_1 before DATE_1.",
  "entities_detected": 3,
  "categories": ["Amount", "Date", "Email"]
}
```

This endpoint mirrors the MCP `pseudonymize` tool: it accepts only `text`, always uses token masking, and does not take a `session_id` in the request body.

### `POST /v1/rehydrate`

Also available at `POST /rehydrate`.

Request body:

```json
{
  "text": "Send AMOUNT_1 to EMAIL_1 before DATE_1."
}
```

Success response:

```json
{
  "text": "Send $500 to alice@example.com before June 1, 2026.",
  "tokens_rehydrated": 3
}
```

Unknown tokens are left unchanged, just like the MCP tool.

### `POST /v1/detect`

Also available at `POST /detect`.

Request body:

```json
{
  "text": "Send $500 to alice@example.com before June 1, 2026."
}
```

Success response:

```json
{
  "entities": [
    {
      "original": "alice@example.com",
      "category": "Email",
      "confidence": 1.0,
      "source": "Pattern"
    },
    {
      "original": "$500",
      "category": "Amount",
      "confidence": 1.0,
      "source": "Financial"
    }
  ]
}
```

Like the MCP tool, this returns raw sensitive substrings in `original`, so use it for review and debugging rather than privacy-preserving outbound traffic.

### `GET` or `POST /v1/vault_stats`

Also available at `GET` or `POST /vault_stats`.

This endpoint takes no parameters and returns safe aggregate stats for the active vault state:

```json
{
  "total_mappings": 4,
  "categories": {
    "AMOUNT": 1,
    "EMAIL": 2,
    "PERSON": 1
  }
}
```

### `POST /v1/configure`

Also available at `POST /configure`.

Request body fields match the MCP tool:

- `profile` (`string | null`)
- `enable` (`string[] | null`)
- `disable` (`string[] | null`)

Example request:

```json
{
  "profile": "legal",
  "enable": ["ip_addresses"]
}
```

Success response:

```json
{
  "active_profile": "legal",
  "secrets": true,
  "financial": true,
  "dates": true,
  "emails": true,
  "phone_numbers": true,
  "ip_addresses": true
}
```

The server applies profile changes first, then `enable`, then `disable`, and rebuilds the detector immediately for future `detect`, `pseudonymize`, chat, and embeddings requests.

### `POST /v1/session_context`

Also available at `POST /session_context`.

Request body:

```json
{
  "session_id": "list"
}
```

If sessions exist, the response matches the MCP tool shape:

```json
{
  "sessions": [
    {
      "session_id": "session-123",
      "message_count": 2,
      "entity_count": 5,
      "coreference_count": 3,
      "sensitivity": "normal",
      "escalation_keywords": [],
      "categories": {
        "Person": 2,
        "Organization": 1,
        "Amount": 2
      },
      "created_at": "2026-05-24T12:00:00+00:00",
      "last_activity": "2026-05-24T12:05:00+00:00"
    }
  ],
  "total": 1
}
```

If the requested session does not exist, the endpoint returns a JSON body like:

```json
{
  "error": "Session 'session-123' not found"
}
```

This endpoint only inspects session state. It does not create sessions by itself; session creation still happens through the chat proxy path when session tracking is enabled and requests include the configured session header.

## Tree endpoints

All tree routes are mounted under `/tree`, not `/v1/tree`.

These endpoints use the `cloakpipe-tree` crate and persist tree JSON files under `tree.storage_path`.

In practice, they are best for:

- building a reusable document index,
- browsing that index,
- retrieving likely sections,
- generating an answer from the indexed structure.

Most tree operations require a working upstream LLM configuration because index construction, tree search, and answer generation rely on model calls.

### Common tree response objects

#### `IndexResponse`

Returned by:

- `POST /tree/index`
- `POST /tree/index/file`
- `GET /tree/{id}`

Fields:

| Field | Type | Meaning |
| --- | --- | --- |
| `id` | `string` | Saved tree identifier |
| `source` | `string` | Source document name or path |
| `description` | `string \| null` | Optional document summary |
| `total_pages` | `number` | Total pages parsed |
| `node_count` | `number` | Total nodes in the tree |
| `max_depth` | `number` | Deepest nesting level |
| `navigation` | `NavigationItem[]` | Flattened navigation map |

#### `NavigationItem`

| Field | Type | Meaning |
| --- | --- | --- |
| `id` | `string` | Node ID |
| `title` | `string` | Node title |
| `summary` | `string \| null` | Optional node summary |
| `depth` | `number` | Nesting depth |
| `pages` | `[number, number]` | Page range |
| `has_children` | `boolean` | Whether the node has child nodes |

### `POST /tree/index`

Use this when your application already has the full document text in memory and wants CloakPipe to build and persist a tree.

Tree index request body:

```json
{
  "name": "msa.txt",
  "text": "Full document text here..."
}
```

Tree index example:

```bash
curl http://127.0.0.1:8900/tree/index \
  -H "Content-Type: application/json" \
  -d '{
    "name": "policy.md",
    "text": "# Refund Policy\n\nCustomers may request refunds within 30 days..."
  }'
```

Tree index response example:

```json
{
  "id": "tree_abc123",
  "source": "policy.md",
  "description": "Refund and exceptions policy",
  "total_pages": 1,
  "node_count": 4,
  "max_depth": 2,
  "navigation": [
    {
      "id": "1",
      "title": "Refund Policy",
      "summary": "Explains refund windows and exceptions.",
      "depth": 1,
      "pages": [1, 1],
      "has_children": true
    }
  ]
}
```

### `POST /tree/index/file`

Use this when the file already exists on the **server's filesystem** and you want CloakPipe to read it directly.

File index request body:

```json
{
  "file_path": "/absolute/or/relative/path/on/the/server"
}
```

#### Important security note

This path is resolved on the machine running CloakPipe, not on the caller's machine. Because of that, this route should only be exposed in trusted environments.

File index example:

```bash
curl http://127.0.0.1:8900/tree/index/file \
  -H "Content-Type: application/json" \
  -d '{"file_path": "./contracts/master-services-agreement.pdf"}'
```

File index success response:

The response shape is the same `IndexResponse` returned by `POST /tree/index`.

#### File index common failure

If `file_path` is missing, the handler returns:

- `400 file_path required`

### `GET /tree/list`

Use this to discover which tree indices are currently saved.

Tree list example:

```bash
curl http://127.0.0.1:8900/tree/list
```

Tree list response example:

```json
[
  {
    "id": "tree_abc123",
    "source": "policy.md",
    "description": "Refund and exceptions policy",
    "total_pages": 1,
    "node_count": 4
  }
]
```

### `GET /tree/{id}`

Use this to retrieve metadata and the navigation map for one saved tree.

Tree get example:

```bash
curl http://127.0.0.1:8900/tree/tree_abc123
```

Tree get success response:

Returns the same `IndexResponse` described above.

#### Tree get failure

If the tree file does not exist, the handler returns `404` with a plain text message beginning with `Tree not found:`.

### `POST /tree/{id}/search`

Use this when you already have a tree ID and want the best-matching nodes for a question.

Tree search request body:

```json
{
  "query": "What is the refund deadline?"
}
```

Tree search example:

```bash
curl http://127.0.0.1:8900/tree/tree_abc123/search \
  -H "Content-Type: application/json" \
  -d '{"query": "What are the termination clauses?"}'
```

Tree search response example:

```json
{
  "node_ids": ["1.2", "1.4"],
  "reasoning": "These sections discuss termination and notice periods.",
  "confidence": 0.92,
  "extracted": [
    {
      "node_id": "1.2",
      "title": "Termination",
      "text": "Explains termination for convenience and for cause.",
      "pages": [7, 9]
    }
  ]
}
```

#### Search result detail

The `extracted[].text` field is built from each node's stored summary, not from the full raw page text. If a node has no summary, `text` can be empty.

### `POST /tree/query`

Use this as the high-level tree endpoint. It can either:

1. search an existing tree, or
2. build a new tree from raw text, save it, then search and answer in one call.

#### Supported request patterns

##### Search an existing tree

```json
{
  "tree_id": "tree_abc123",
  "query": "What is the indemnity cap?"
}
```

##### Build from text, then answer

```json
{
  "name": "nda.txt",
  "text": "Full document text here...",
  "query": "When does confidentiality expire?"
}
```

#### Query flow

1. Loads a saved tree if `tree_id` is present.
2. Otherwise builds a new tree from `text` and optional `name`.
3. Searches the tree.
4. Builds a context block from matched node summaries.
5. Calls the upstream chat-completions API to generate a final answer.

Tree query response example:

```json
{
  "answer": "The agreement caps indemnity at the total fees paid in the prior 12 months.",
  "sources": [
    {
      "node_id": "2.1",
      "title": "Indemnity",
      "text": "Caps liability and indemnity obligations.",
      "pages": [12, 13]
    }
  ],
  "tree_id": "tree_abc123",
  "reasoning": "The indemnity section directly addresses caps and exclusions."
}
```

#### Query context detail

The current implementation builds the answer context from node summaries, not from full extracted raw passages. That makes this endpoint useful for structured document QA, but it is not yet a quote-accurate passage extraction endpoint.

#### Query validation failure

If neither `tree_id` nor `text` is provided, the handler returns:

- `400 Either tree_id or text required`

### `DELETE /tree/{id}`

Use this to remove a saved tree JSON file.

Tree delete example:

```bash
curl -X DELETE http://127.0.0.1:8900/tree/tree_abc123
```

Tree delete response example:

```json
{
  "deleted": "tree_abc123"
}
```

If the tree does not exist, the handler returns `404` with a plain text message beginning with `Tree not found:`.

## Session endpoints

Sessions are disabled by default. To make these endpoints useful, enable them in `cloakpipe.toml`:

```toml
[session]
enabled = true
id_from = "header:x-session-id"
ttl_seconds = 1800
coreference = true
sensitivity_escalation = true
session_threshold = 0.8
```

Sessions are created from chat requests, not from direct calls to `/sessions`.

The session APIs expose only safe statistics. They do **not** return raw PII.

### `GET /sessions`

Use this to list active session summaries.

Sessions list example:

```bash
curl http://127.0.0.1:8900/sessions
```

Sessions list response example:

```json
[
  {
    "session_id": "demo-session-1",
    "message_count": 3,
    "entity_count": 5,
    "coreference_count": 4,
    "sensitivity": "normal",
    "escalation_keywords": [],
    "categories": {
      "Person": 2,
      "Email": 1,
      "Organization": 2
    },
    "created_at": "2026-05-24T12:00:00+00:00",
    "last_activity": "2026-05-24T12:03:10+00:00"
  }
]
```

### `GET /sessions/{id}`

Use this to inspect one session summary in more detail.

Session inspect example:

```bash
curl http://127.0.0.1:8900/sessions/demo-session-1
```

Session inspect success response:

Returns one `SessionStats` object with the same fields shown above.

#### Session inspect failure

If the session does not exist, the handler returns:

- `404 Session <id> not found`

### `DELETE /sessions/{id}`

Use this to flush one session explicitly.

Session flush example:

```bash
curl -X DELETE http://127.0.0.1:8900/sessions/demo-session-1
```

Session flush response example:

```json
{
  "flushed": true,
  "session_id": "demo-session-1"
}
```

If the session is missing, the endpoint still returns JSON, but `flushed` is `false`.

### `DELETE /sessions`

Use this to flush every session currently stored in memory.

Flush-all example:

```bash
curl -X DELETE http://127.0.0.1:8900/sessions
```

Flush-all response example:

```json
{
  "flushed": 3
}
```

### Session TTL caveat

The session code defines TTL logic, but the current HTTP layer does not run a periodic eviction task. In practice, that means you should use the flush endpoints if you need deterministic cleanup.

## Recommended usage patterns

### Standard chat proxy

Use `POST /v1/chat/completions` when you want drop-in privacy protection for OpenAI-style chat calls.

Recommended approach:

1. Point your SDK base URL at CloakPipe.
2. Keep your real upstream API key on the CloakPipe server.
3. Send a stable `x-session-id` if you want consistent cross-message pseudonyms.
4. Use non-streaming mode if you want the extra leaked-PII output scan.

### Embeddings for indexing or retrieval

Use `POST /v1/embeddings` when you want PII masking before embedding generation.

Recommended approach:

1. Keep `input` as a string or array of strings.
2. Do not assume vector encryption happens in the HTTP layer today.
3. Treat the result as a normal upstream embeddings response plus a request ID header.

### Tree workflow for reusable document search

Use this pattern when documents live longer than one request:

1. `POST /tree/index` or `POST /tree/index/file`
2. `GET /tree/list` to discover saved IDs later
3. `GET /tree/{id}` to inspect structure
4. `POST /tree/{id}/search` for retrieval-only flows
5. `POST /tree/query` for retrieval plus final answer generation
6. `DELETE /tree/{id}` when you want cleanup

Prefer `POST /tree/index` over `POST /tree/index/file` for remote callers. It is safer because you send content directly instead of exposing server-local file access.

### One-shot document Q&A

Use `POST /tree/query` with `text` when:

- you have document text in hand,
- you want CloakPipe to save the resulting tree automatically,
- you want one answer back immediately.

This is the simplest tree entry point for app developers.

## What is not in the current server

The current router does **not** register several endpoints mentioned in older docs and design notes.

These are **not** present in the current `main` branch router:

- `/v1/completions`
- wildcard passthrough routes such as `/*`
- `/v1/tree/build`
- `/v1/tree/search`
- `/v1/tree/list`
- `/_cloakpipe/stats`
- `/_cloakpipe/mode`

If you are integrating against the current source tree, use the routes documented in this file.
